【My Neovim Config】

(written with nvim)

単一ファイルの設定
March 30, 2026 Dev Tools

「Simplicity is the ultimate sophistication」

Real talk: my Neovim config is one file. init.lua. 744 lines. That's it. No fancy folder structure, no "modular architecture," no splitting things into 20 different files because some tutorial told me to. Just one file that loads faster than my coffee brews ☕

1. Why One File? The Performance is Insane

Okay so everyone and their grandma says to modularize your config. Put plugins in lua/plugins/, keymaps somewhere else, options in another file. But here's the tea: every single require() is a disk read.

On a fast NVMe? Whatever, negligible. But why waste I/O when you don't have to? My single-file approach hits the disk once. That's it. Neovim opens instantly. No cap.

Plus, and this is the real flex: I know exactly where everything is. Need to tweak my LSP? Scroll down. Keymaps? Search "KEYMAPS." There's no "wait which file did I put that in" moment. One scrollable doc, everything visible, brain doesn't have to context switch. It's giving efficiency.

Here's the vibe of my 744 lines:

-- ==========================================
-- SECTION 1: Options (set immediately, no delays)
-- ==========================================
vim.opt.mouse = ""
vim.opt.relativenumber = true
vim.opt.conceallevel = 2
vim.opt.clipboard = "unnamedplus"
-- ... more options

-- ==========================================
-- SECTION 2: Plugins (vim.pack, built into nvim 0.12)
-- ==========================================
-- No external plugin manager needed, we're built different
vim.pack.add({
  { src = "https://github.com/neovim/nvim-lspconfig" },
  { src = "https://github.com/Saghen/blink.cmp" },
  -- ...
})

-- ==========================================
-- SECTION 3: LSP (deferred, loads on file open)
-- ==========================================
-- Smart loading - doesn't slow startup

-- ==========================================
-- SECTION 4: Keymaps
-- ==========================================
local map = vim.keymap.set
map("n", "<leader>ff", function() fzf_lazy.files() end)
-- ...

-- ==========================================
-- SECTION 5: Colorscheme
-- ==========================================
vim.cmd("colorscheme kanagawa")

-- ==========================================
-- SECTION 6: Custom Capture Module (~240 lines)
-- ==========================================
-- My note-taking system, inlined to avoid file I/O
-- Yes I wrote this myself, yes it slaps

Clear sections, logical flow, zero unnecessary requires. Main character energy.

2. The Plugin Lineup (Curated, Not Cluttered)

I don't have 50 plugins like some people's configs that take 3 seconds to load. I have like 15, and most are lazy-loaded. Here's what actually matters:

The Essentials (Load Immediately)

vim.pack - Built into Neovim 0.12. No external dependency, no extra lua files to require, just pure native speed. Ditched lazy.nvim because why use a plugin to manage plugins when nvim has it built-in now? We're not doing extra work in this house.

nvim-lspconfig + mason-lspconfig - LSP for bash, C/C++, Lua, Markdown. But here's the smart part: it doesn't load until you actually open a file. Deferred via autocmd so startup stays fast. Big brain moves only.

blink.cmp - Completion that just works. Fast, looks clean, integrates with mini.icons for those satisfying file type icons. The mini.icons integration especially? *Chef's kiss*

mini.nvim - The ultimate Swiss Army knife. Statusline, autopairs, file explorer, icons, AND mini.clue for which-key hints. One plugin, five problems solved. Minimalism wins.

Kanagawa - My colorscheme. Warm tones, easy on the eyes for those 4-hour coding sessions. It's not just an aesthetic choice, it's a productivity choice. The vibe is immaculate.

The Heavy Lifters (Lazy Loaded, Obviously)

fzf-lua - Fuzzy finding for literally everything. Files, grep, buffers, help tags. But it only loads when I hit <leader>ff. Before that? Doesn't exist in memory. Ghost mode activated.

-- Lazy loaded fzf-lua using metatable magic
-- This is probably the cleanest lazy-loading trick I know
local fzf_lazy = setmetatable({}, {
  __index = function(t, k)
    local fzf = require("fzf-lua")
    if not vim.g.fzf_setup_done then
      fzf.setup({ winopts = { height = 0.7, width = 0.8 } })
      vim.g.fzf_setup_done = true
    end
    rawset(t, k, fzf[k])
    return fzf[k]
  end,
})

Obsidian + render-markdown - My note-taking workflow. Only loads for markdown files. Why have markdown plugins resident in memory when I'm editing Lua or C? That's just wasteful. Only pay for what you use.

conform.nvim - Formatting with prettier, stylua, clang-format. Loaded on demand via vim.defer_fn. Your code gets formatted, your startup time doesn't suffer. Best of both worlds.

nvim-treesitter - Syntax highlighting that actually understands your code. Deferred 100ms after startup so it doesn't block the editor from opening. Async loading is the way.

The Custom Stuff (I Wrote This)

I have a 240-line "Capture" module written directly in my init.lua. It's a note-taking system with floating windows for quick capture to my Obsidian vault. Could I have put this in lua/capture.lua? Technically yes. But then I'd have:

  • Another file to keep track of
  • Another require() slowing startup
  • Another context switch when I want to edit it
  • More cognitive load

Instead it's right there in my init.lua. Functions for finding recent notes, opening floating windows, inserting into sections (Log, Todo, Learned), Tab navigation between sections. All inline. All fast. All understood.

3. Why Neovim? The Real Reasons

After trying VS Code, Sublime, JetBrains IDEs, and basically everything else, why settle on Neovim?

It's transparent. I can read every single line of my 744-line config and understand exactly what it does. No magic, no hidden behaviors, no "helpful" abstractions that actually just get in the way. If it breaks, I know why. If I want to change it, I know where.

It's fast. My config loads in literal milliseconds. Not because I'm some genius programmer, but because I eliminated unnecessary file I/O and aggressively deferred everything that isn't needed immediately. Performance by design.

It's consistent. Same editor on my laptop, on a remote server via SSH, in a Docker container, on a VPS somewhere. I never have to context switch between different environments or learn new keybindings. Muscle memory is preserved everywhere.

It respects my flow. No popups interrupting me. No AI autocomplete making suggestions I didn't ask for. No "smart" features trying to predict what I want. Just me, the text, and the tools I chose to enable. Silence is underrated.

The Bottom Line

One file. Curated plugins. Aggressive lazy-loading. Total understanding of your tools.

When something breaks (rare), I know exactly where to look. When I want to change something, I know exactly where to find it. When I open Neovim, it just works, instantly, every time. That reliability? That speed? That understanding? That's the goal.

Is this setup for everyone? Honestly, no. Some people prefer the organization of modular configs, the hand-holding of heavier plugin managers, the "batteries included" approach where everything is configured for you. And that's totally valid. Different workflows for different people.

But if you value understanding your tools at a fundamental level, if you believe software should get out of your way and just work, if you're willing to trade a bit of convenience for a lot of control and speed—try the single-file approach. Start small, keep it simple, build it up. You might be surprised how liberating it is to truly own your editor configuration.
