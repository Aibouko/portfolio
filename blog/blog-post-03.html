【Why I Keep My Neovim Config in One File】

単一ファイルの設定
March 30, 2026 Dev Tools

「Simplicity is the ultimate sophistication」

I've been using Neovim for a while now, and one question I keep getting is: "Why do you keep everything in a single init.lua instead of splitting it into multiple files?" Today I'll explain my philosophy, the plugins that power my workflow, and why Neovim has become my editor of choice.

1. Why One File? The Beauty of init.lua

Most Neovim users eventually split their config into a modular structure: plugins.lua, keymaps.lua, settings.lua, and so on. It's the recommended approach. So why do I resist?

It's all about cognitive load. When I open my config, I want to see everything at once. No jumping between files, no remembering where I put that specific keymap, no mental context switching. Just one scrollable document where I can find anything in seconds.

Here's how I organize my single file:

-- ==========================================
-- NEOVIM CONFIG - Everything in one place
-- ==========================================

-- SECTION 1: Options & Settings
local opt = vim.opt
opt.number = true
opt.relativenumber = true
opt.tabstop = 2
opt.shiftwidth = 2

-- SECTION 2: Keymaps
local map = vim.keymap.set
map('n', '<leader>ff', ':Telescope find_files<CR>')

-- SECTION 3: Plugin Setup
-- (Using lazy.nvim for plugin management)

With clear section comments and consistent organization, my ~300 line init.lua is actually easier to navigate than a scattered modular setup. When something breaks, I know exactly where to look.

2. My Essential Plugins

I don't use many plugins. I believe in a minimal setup that gets out of my way. Here are the ones I actually need:

File Finding & Navigation

Telescope is my window into the project. Fuzzy finding files, grepping code, searching buffers - it's all there.

-- Telescope for fuzzy finding
{
  'nvim-telescope/telescope.nvim',
  dependencies = { 'nvim-lua/plenary.nvim' },
  config = function()
    local builtin = require('telescope.builtin')
    vim.keymap.set('n', '<leader>ff', builtin.find_files, {})
    vim.keymap.set('n', '<leader>fg', builtin.live_grep, {})
  end
}

Language Support

nvim-treesitter for better syntax highlighting and code understanding. It makes every language feel premium.

nvim-lspconfig for Language Server Protocol support. Go to definition, hover documentation, diagnostics - the IDE experience without the IDE bloat.

The Aesthetic Touch

Catppuccin theme. I use the mocha variant - warm, easy on the eyes, perfect for long coding sessions. It matters more than you'd think.

-- Catppuccin theme setup
{
  "catppuccin/nvim",
  name = "catppuccin",
  priority = 1000,
  config = function()
    require("catppuccin").setup({
      flavour = "mocha",
    })
    vim.cmd.colorscheme "catppuccin"
  end
}

Status Line

lualine.nvim - Clean, minimal, shows exactly what I need: mode, filename, git branch, line numbers. No fluff.

3. Why Neovim?

After using VS Code, Sublime Text, and various JetBrains products, why did I settle on Neovim?

It's fast. No Electron bloat, no startup delays. It opens instantly, even on large projects.

It's consistent. The same editor on my laptop, on a remote server, in a Docker container. I SSH somewhere and my familiar environment is there.

It's mine. Every keymap, every behavior, every visual choice is intentional. I understand every line of my config because I wrote it. When something breaks, I know exactly why.

It stays out of my way. No popups, no "helpful" suggestions interrupting flow, no AI autocomplete getting cocky. Just me and the text.

The Verdict

One file. Essential plugins. Total control. That's my Neovim setup, and I wouldn't have it any other way.

Is it for everyone? No. But if you value simplicity, speed, and understanding your tools at a fundamental level, consider giving the single-file approach a try. You might be surprised how liberating it is.
