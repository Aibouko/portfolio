【My 744-Line Neovim Config】

単一ファイルの設定
March 30, 2026 Dev Tools

「Simplicity is the ultimate sophistication」

My Neovim configuration lives in a single file: init.lua. It's 744 lines, includes everything from keymaps to LSP setup to a custom note-taking system, and it loads blazingly fast. People ask why I don't split it. Let me explain.

1. Why One File? Speed and Sanity

Everyone says to modularize. Put plugins in lua/plugins/, keymaps in lua/keymaps.lua, options in lua/options.lua. But here's the thing: every require() is a file system call.

When Neovim starts, it has to hit the disk for each module. On a fast SSD that's negligible, but on slower systems or when you're opening files constantly, it adds up. My single-file approach eliminates all that I/O.

More importantly: I know exactly where everything is. Need to tweak my LSP settings? Scroll down. Want to add a keymap? Search for "KEYMAPS". There's no mental overhead of remembering which file contains what.

Here's how I organize my 744 lines:

-- ==========================================
-- SECTION 1: Options (fastest - set immediately)
-- ==========================================
vim.opt.mouse = ""
vim.opt.relativenumber = true
vim.opt.conceallevel = 2
vim.opt.clipboard = "unnamedplus"
-- ... more options

-- ==========================================
-- SECTION 2: Plugin Management (vim.pack)
-- ==========================================
-- Using Neovim 0.11+ built-in plugin manager
vim.pack.add({
  { src = "https://github.com/neovim/nvim-lspconfig" },
  { src = "https://github.com/Saghen/blink.cmp" },
  -- ...
})

-- ==========================================
-- SECTION 3: LSP Configuration
-- ==========================================
-- Deferred setup - only loads when opening files

-- ==========================================
-- SECTION 4: Keymaps
-- ==========================================
local map = vim.keymap.set
map("n", "<leader>ff", function() fzf_lazy.files() end)
-- ...

-- ==========================================
-- SECTION 5: Colorscheme
-- ==========================================
vim.cmd("colorscheme kanagawa")

-- ==========================================
-- SECTION 6: Custom Capture Module (~240 lines)
-- ==========================================
-- My note-taking system, inlined to avoid file I/O

Clear sections, logical flow, and zero require() calls slowing things down.

2. What I Actually Use

I don't have 50 plugins. I have maybe 15, and half are lazy-loaded. Here's what actually matters:

The Essentials (Immediate Load)

vim.pack - Neovim 0.11+'s built-in plugin manager. No external dependency, no lua files to require, just pure speed. It replaced lazy.nvim for me because I don't need the complexity.

nvim-lspconfig + mason-lspconfig - LSP setup for bash, C/C++, Lua, and Markdown. But here's the trick: it doesn't load until you open a file. Deferred via autocmd.

blink.cmp - Completion that just works. Fast, pretty, integrates with mini.icons for those sweet file type icons.

mini.nvim - The Swiss Army knife. Statusline, file pairs, file explorer, icons, and mini.clue for which-key-style hints. One plugin, five problems solved.

Kanagawa - My colorscheme. Warm, easy on the eyes, perfect for long sessions. It's not just pretty; it's functional.

The Heavy Lifters (Lazy Loaded)

fzf-lua - Fuzzy finding for files, grep, buffers, everything. Loaded only when I hit <leader>ff. Before that, it doesn't exist in memory.

-- Lazy loaded fzf-lua using metatable magic
local fzf_lazy = setmetatable({}, {
  __index = function(t, k)
    local fzf = require("fzf-lua")
    if not vim.g.fzf_setup_done then
      fzf.setup({ winopts = { height = 0.7, width = 0.8 } })
      vim.g.fzf_setup_done = true
    end
    rawset(t, k, fzf[k])
    return fzf[k]
  end,
})

Obsidian + render-markdown - My note-taking system. Only loads for markdown files. Why have markdown plugins eating memory when I'm editing Lua?

conform.nvim - Formatting with prettier, stylua, clang-format. Loaded when needed, not before.

nvim-treesitter - Syntax highlighting that doesn't suck. Deferred 100ms after startup so it doesn't block.

The Custom Bits

I wrote a 240-line "Capture" module directly in my init.lua. It's a note-taking system that opens floating windows for quick capture to my Obsidian vault. No separate file, no require(), just inline Lua functions that handle:

  • Finding recent notes
  • Opening floating capture windows
  • Inserting into specific sections (Log, Todo, Learned)
  • Tab completion between sections

Could I have put this in lua/capture.lua? Sure. But then I'd have another file to track, another require() to slow startup, another context switch when editing.

3. Why Neovim?

After VS Code, Sublime, and various JetBrains IDEs, why Neovim?

It's transparent. I can read every line of my 744-line config and understand exactly what it does. No magic, no hidden behaviors, no "helpful" abstractions getting in my way.

It's fast. My config loads in milliseconds. Not because I'm a genius, but because I eliminated unnecessary file I/O and deferred everything that isn't immediately needed.

It's consistent. Same editor on my laptop, on a remote server via SSH, in a Docker container, on a VPS. I never have to context switch between different environments.

It gets out of the way. No popups interrupting my flow. No AI autocomplete making suggestions I didn't ask for. No "smart" features trying to predict what I want. Just text and me.

The Philosophy

One file. Minimal plugins. Aggressive lazy-loading. Total understanding.

When something breaks, I know where to look. When I want to change something, I know where to find it. When I open Neovim, it just works, instantly.

Is it for everyone? No. Some people prefer the organization of modular configs, the hand-holding of heavier plugin managers, the "batteries included" approach of VS Code. And that's fine.

But if you value understanding your tools at a fundamental level, if you believe that software should get out of your way and just work, if you're willing to trade a bit of convenience for a lot of control—try the single-file approach. Start small. Keep it simple. You might be surprised how liberating it is.
